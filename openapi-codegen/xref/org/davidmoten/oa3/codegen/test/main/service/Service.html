25   * <p>EGC API
26   * <h1>EGC API</h1>
27   * <h2>Design considerations</h2>
28   * <h3>Two-phase send</h3>
29   * <p>Sending a new MSI is two-phase (create then send). This is
30   * because the msiId value is a critical piece of information
31   * for the client (to be able to check status and/or cancel)
32   * and the API design must ensure that the client is safely
33   * in possession of msiID before a send happens. If the
34   * method was one phase then there would be no guarantee that
35   * the msiID was received (a failure could happen in any link
36   * in the return snetworking chain like a router, firewall, proxy
37   * server or indeed a problem at the client application end
38   * that prevented persistence of the msiID for later use).
39   * Moreover, if the method was one phase and a failure in the
40   * network chain occurred then not only would an orphan MSI
41   * be sent by the provider but the client would not have
42   * knowledge that the MSI had been successfully queued for
43   * sending and would naturally retry the send (multiple times
44   * even) and we end up with the same MSI being sent 2+ times.
45   * <p>To further clarify the problem being solved by a two phase
46   * send here is a <a href="https://stackoverflow.com/questions/49444723/how-to-make-sure-the-http-response-was-delivered">discussion</a>
47   * of the delivery guarantees of HTTP responses.
48   * <p>Note that a one phase call where the client generated a new
49   * unique id (using a UUID for instance) is a possible solution
50   * but is less desirable because it introduces a problematic
51   * edge case where the client accidentally uses the same id more
52   * than once. If two different messages are sent with the same id
53   * (concurrently even) then the service should ensure that only
54   * one message is accepted and that the service consumer is aware
55   * that the other message failed. To do this demands coordination
56   * with a single transactional resource (like a relational database) which
57   * also demands that that resource is highly available (relational
58   * databases are often not great at that during upgrade cycles). There
59   * are ways to get high availability (highly available cloud services like
60   * DynamoDB and many more can offer conditional updates) but there is
61   * a much simpler way with two-phase.
62   * <p>If instead of the one-phase call the server creates the msiId and communicates
63   * it to the client then the server side can potentially be scaled
64   * with ease if the msiID is a <a href="https://en.wikipedia.org/wiki/Universally_unique_identifier">UUID</a>
65   * for instance (which is effectively unique without coordination
66   * with other nodes).
67   * <p>For example, a highly available and scalable
68   * service could be constructed in AWS cloud using API Gateway with
69   * Lambda integrations that for the create message and send actions
70   * does this
71   * <ul>
72   * <li><strong>Create:</strong> Generate a new UUID, place the message content and UUID on
73   * to a queue for processing , return the UUID</li>
74   * <li><strong>Send:</strong> Place the UUID parameter on to a queue for processing</li>
75   * </ul>
76   * <p>A separate component then actions items on the queue(s). When both
77   * the send and create messages have been read then an actual send can take
78   * place. What is clear from this design is that many concurrent nodes could
79   * be receiving messages without coordinating with a central node/service
80   * to ensure id uniqueness.
81   * <p>Note also that to support two-phase send the status value of
82   * <code>CREATED</code> is included.
83   * <h3>Pagination</h3>
84   * <p>The List MSIs action uses a paginated response as the number of
85   * MSIs in a response can get large. Pagination can reduce server
86   * overhead and improve response times. Client-driven pagination is
87   * where the client specifies an offset (skip) field and that number
88   * of rows is skipped by the server to return the next page. This
89   * can be inefficient for the server-side (see <a href="https://use-the-index-luke.com/no-offset">discussion</a>) and it is preferred to
90   * use server-driven pagination which is where each page returned
91   * also includes a continuation token to be included in the next page
92   * call. The nice thing about this approach is that the server side
93   * can simply return an offset in the continuation token if desired but
94   * we enable more efficient techniques if wanted later.
95   * <h3>Client specific identifiers</h3>
96   * <p>Early versions of this API have suggested the inclusion of a
97   * <code>NationalSASId</code> field in the created MSI with the purpose of allowing
98   * a client to correlate an MSI with its internal data.
99   * <p>This field is a convenience only and thus theoretically should not be
100  * included. A client should manage its correlations itself by storing the
101  * unique msiId returned by the service paired with its internal identifiers.
102  * <p>If something is required then it should be labelled something
103  * like <code>tag</code> and have arbitrary values so that the client can use it for
104  * anything. Labelling it <code>NationalSASId</code> suggests more meaning to the
105  * field than it may have. <strong>TODO</strong> confirm.
106  * <h3>Geometry</h3>
107  * <p>Note that the api below allows for float precision locations for
108  * geographic circles and rectangles. An implementation of this API
109  * may choose to use the location with reduced precision (for example
110  * lat longs rounded to nearest integer).
111  * <h3>Cancellation</h3>
112  * <p>A PUT to an <code>/msi/[id}</code> path ]with content like <code>{&quot;isCancelled&quot;:true}</code>
113  * has been suggested as a way of cancelling a broadcast. This can be
114  * achieved in a much simpler way with the DELETE verb without content
115  * (a cancel action can be considered as a logical delete in the context
116  * of this API). A cancelled broadcast cannot be changed in status but
117  * can be queried.
118  * <h3>Abstraction of C-Codes</h3>
119  * <p>Initial proposals for the API suggested a partial abstraction of C-Codes.
120  * In particular Priority, MsiType and AreaType were abstracted. This API
121  * demonstrates a full abstraction of C-Codes. It is equivalent to C-Codes
122  * but has an easier to read and process representation and the mapping to C-Codes
123  * then becomes a server-side implementation detail. By using the data modelling
124  * constructs of OpenAPI v3 and JSON Schema users can generate code for their
125  * API-consuming application that imposes compile-time checking (varies on language)
126  * instead of experiencing runtime failures.
127  * <p><strong>TODO</strong> is there a requirement for full explicit C-Code support (zero
128  * abstraction)?
129  * <h3>Auth</h3>
130  * <p>Initial proposals for this API included a re-authenticate
131  * method whereby a new token was returned if a currently valid token was
132  * supplied. This is a security hole in that anyone in possession of one
133  * valid token (but not the username and password) can stay authenticated
134  * forever. In the same vein, a sensible limit on validity duration of a
135  * token should be imposed so that a leaked token cannot be used for long.
136  * Given the likely usage of the API (in terms of the number of calls made
137  * in a time interval by a client) there should be no significant performance
138  * penalty forcing a refresh of the token each hour (or even less).
139  * <p>Bearer authentication is used (<a href="https://www.rfc-editor.org/rfc/rfc6750.html#page-5">RFC6750</a>).
140  * Bearer authentication is carried in the <code>Authorization</code> request header in this format:
141  * <pre><code>Authorization: Bearer BASE64_ENCODED_TOKEN
142  * </code></pre>
143  * <p><strong>TODO</strong> The encoded content of the token is not defined (for example, JWT could be used)
144  * but is left to the implementer. Should the authentication flow follow an existing
145  * standard like OAuth 2.0?
146  * <h3>Naming</h3>
147  * <p>Previous API drafts used the field names <code>startDate</code> and <code>endDate</code> for an MSI.
148  * Given that those fields refer to timestamps not just dates the names <code>startTime</code>
149  * and <code>endTime</code> have been used.
150  * <h3>Acknowledgments</h3>
151  * <p>Iridium can at times provide receive and read acknowledgements. <strong>TODO</strong> get
152  * better documentation of the capability and a proper specification for their
153  * response from a list acks call.
154  * <h3>Timings</h3>
155  * <p>There may be use cases for the following additional fields on an MSI:
156  * <ul>
157  * <li>createdTime</li>
158  * <li>cancelledTime</li>
159  * </ul>
160  * <p>A user could record in their own systems when they created or cancelled a broadcast
161  * but it might help problem diagnosis if that information was together.
162  * <p><strong>TODO</strong> confirm
163  * <p>It would also be useful in some circumstances for users to know exactly when a
164  * message was broadcast by satellite. Although messages may be scheduled for
165  * immediate or later broadcast there may be significant delays till the broadcast
166  * occurs and the user should be able to see the actual broadcast times.
167  * <p><strong>TODO</strong> discuss with satellite providers
168  */
