31   * <p>EGC API
32   * <h1>EGC API</h1>
33   * <h2>Design considerations</h2>
34   * <h3>Two-phase send</h3>
35   * <p>Sending a new MSI is two-phase (create then send). This is
36   * because the msiId value is a critical piece of information
37   * for the client (to be able to check status and/or cancel)
38   * and the API design must ensure that the client is safely
39   * in possession of msiID before a send happens. If the
40   * method was one phase then there would be no guarantee that
41   * the msiID was received (a failure could happen in any link
42   * in the return snetworking chain like a router, firewall, proxy
43   * server or indeed a problem at the client application end
44   * that prevented persistence of the msiID for later use).
45   * Moreover, if the method was one phase and a failure in the
46   * network chain occurred then not only would an orphan MSI
47   * be sent by the provider but the client would not have
48   * knowledge that the MSI had been successfully queued for
49   * sending and would naturally retry the send (multiple times
50   * even) and we end up with the same MSI being sent 2+ times.
51   * <p>To further clarify the problem being solved by a two phase
52   * send here is a <a href="https://stackoverflow.com/questions/49444723/how-to-make-sure-the-http-response-was-delivered">discussion</a>
53   * of the delivery guarantees of HTTP responses.
54   * <p>Note that a one phase call where the client generated a new
55   * unique id (using a UUID for instance) is a possible solution
56   * but is less desirable because it introduces a problematic
57   * edge case where the client accidentally uses the same id more
58   * than once. If two different messages are sent with the same id
59   * (concurrently even) then the service should ensure that only
60   * one message is accepted and that the service consumer is aware
61   * that the other message failed. To do this demands coordination
62   * with a single transactional resource (like a relational database) which
63   * also demands that that resource is highly available (relational
64   * databases are often not great at that during upgrade cycles). There
65   * are ways to get high availability (highly available cloud services like
66   * DynamoDB and many more can offer conditional updates) but there is
67   * a much simpler way with two-phase.
68   * <p>If instead of the one-phase call the server creates the msiId and communicates
69   * it to the client then the server side can potentially be scaled
70   * with ease if the msiID is a <a href="https://en.wikipedia.org/wiki/Universally_unique_identifier">UUID</a>
71   * for instance (which is effectively unique without coordination
72   * with other nodes).
73   * <p>For example, a highly available and scalable
74   * service could be constructed in AWS cloud using API Gateway with
75   * Lambda integrations that for the create message and send actions
76   * does this
77   * <ul>
78   * <li><strong>Create:</strong> Generate a new UUID, place the message content and UUID on
79   * to a queue for processing , return the UUID</li>
80   * <li><strong>Send:</strong> Place the UUID parameter on to a queue for processing</li>
81   * </ul>
82   * <p>A separate component then actions items on the queue(s). When both
83   * the send and create messages have been read then an actual send can take
84   * place. What is clear from this design is that many concurrent nodes could
85   * be receiving messages without coordinating with a central node/service
86   * to ensure id uniqueness.
87   * <p>Note also that to support two-phase send the status value of
88   * <code>CREATED</code> is included.
89   * <h3>Pagination</h3>
90   * <p>The List MSIs action uses a paginated response as the number of
91   * MSIs in a response can get large. Pagination can reduce server
92   * overhead and improve response times. Client-driven pagination is
93   * where the client specifies an offset (skip) field and that number
94   * of rows is skipped by the server to return the next page. This
95   * can be inefficient for the server-side (see <a href="https://use-the-index-luke.com/no-offset">discussion</a>) and it is preferred to
96   * use server-driven pagination which is where each page returned
97   * also includes a continuation token to be included in the next page
98   * call. The nice thing about this approach is that the server side
99   * can simply return an offset in the continuation token if desired but
100  * we enable more efficient techniques if wanted later.
101  * <h3>Client specific identifiers</h3>
102  * <p>Early versions of this API have suggested the inclusion of a
103  * <code>NationalSASId</code> field in the created MSI with the purpose of allowing
104  * a client to correlate an MSI with its internal data.
105  * <p>This field is a convenience only and thus theoretically should not be
106  * included. A client should manage its correlations itself by storing the
107  * unique msiId returned by the service paired with its internal identifiers.
108  * <p>If something is required then it should be labelled something
109  * like <code>tag</code> and have arbitrary values so that the client can use it for
110  * anything. Labelling it <code>NationalSASId</code> suggests more meaning to the
111  * field than it may have. <strong>TODO</strong> confirm.
112  * <h3>Geometry</h3>
113  * <p>Note that the api below allows for float precision locations for
114  * geographic circles and rectangles. An implementation of this API
115  * may choose to use the location with reduced precision (for example
116  * lat longs rounded to nearest integer).
117  * <h3>Cancellation</h3>
118  * <p>A PUT to an <code>/msi/[id}</code> path ]with content like <code>{&quot;isCancelled&quot;:true}</code>
119  * has been suggested as a way of cancelling a broadcast. This can be
120  * achieved in a much simpler way with the DELETE verb without content
121  * (a cancel action can be considered as a logical delete in the context
122  * of this API). A cancelled broadcast cannot be changed in status but
123  * can be queried.
124  * <h3>Abstraction of C-Codes</h3>
125  * <p>Initial proposals for the API suggested a partial abstraction of C-Codes.
126  * In particular Priority, MsiType and AreaType were abstracted. This API
127  * demonstrates a full abstraction of C-Codes. It is equivalent to C-Codes
128  * but has an easier to read and process representation and the mapping to C-Codes
129  * then becomes a server-side implementation detail. By using the data modelling
130  * constructs of OpenAPI v3 and JSON Schema users can generate code for their
131  * API-consuming application that imposes compile-time checking (varies on language)
132  * instead of experiencing runtime failures.
133  * <p><strong>TODO</strong> is there a requirement for full explicit C-Code support (zero
134  * abstraction)?
135  * <h3>Auth</h3>
136  * <p>Initial proposals for this API included a re-authenticate
137  * method whereby a new token was returned if a currently valid token was
138  * supplied. This is a security hole in that anyone in possession of one
139  * valid token (but not the username and password) can stay authenticated
140  * forever. In the same vein, a sensible limit on validity duration of a
141  * token should be imposed so that a leaked token cannot be used for long.
142  * Given the likely usage of the API (in terms of the number of calls made
143  * in a time interval by a client) there should be no significant performance
144  * penalty forcing a refresh of the token each hour (or even less).
145  * <p>Bearer authentication is used (<a href="https://www.rfc-editor.org/rfc/rfc6750.html#page-5">RFC6750</a>).
146  * Bearer authentication is carried in the <code>Authorization</code> request header in this format:
147  * <pre><code>Authorization: Bearer BASE64_ENCODED_TOKEN
148  * </code></pre>
149  * <p><strong>TODO</strong> The encoded content of the token is not defined (for example, JWT could be used)
150  * but is left to the implementer. Should the authentication flow follow an existing
151  * standard like OAuth 2.0?
152  * <h3>Naming</h3>
153  * <p>Previous API drafts used the field names <code>startDate</code> and <code>endDate</code> for an MSI.
154  * Given that those fields refer to timestamps not just dates the names <code>startTime</code>
155  * and <code>endTime</code> have been used.
156  * <h3>Acknowledgments</h3>
157  * <p>Iridium can at times provide receive and read acknowledgements. <strong>TODO</strong> get
158  * better documentation of the capability and a proper specification for their
159  * response from a list acks call.
160  * <h3>Timings</h3>
161  * <p>There may be use cases for the following additional fields on an MSI:
162  * <ul>
163  * <li>createdTime</li>
164  * <li>cancelledTime</li>
165  * </ul>
166  * <p>A user could record in their own systems when they created or cancelled a broadcast
167  * but it might help problem diagnosis if that information was together.
168  * <p><strong>TODO</strong> confirm
169  * <p>It would also be useful in some circumstances for users to know exactly when a
170  * message was broadcast by satellite. Although messages may be scheduled for
171  * immediate or later broadcast there may be significant delays till the broadcast
172  * occurs and the user should be able to see the actual broadcast times.
173  * <p><strong>TODO</strong> discuss with satellite providers
174  */
